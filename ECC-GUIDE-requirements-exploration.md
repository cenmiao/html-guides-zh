# ECC 需求理解与探索阶段 - 详细指南

> 本文档详细说明 ECC 在需求理解与探索阶段的使用方法

---

## 核心原则

ECC 在需求理解与探索阶段遵循 **"Research Before You Code"**（先研究再编码）的核心原则。

**阶段目标：**
1. 理解项目背景 - 技术栈、架构、现有代码结构
2. 研究现有方案 - 避免重复造轮子
3. 明确需求边界 - 识别风险和依赖
4. 制定实现蓝图 - 分解任务、规划步骤

---

## 四大核心 Skill

### 1. `/ecc:plan` - 实现规划命令

**用途：** 在任何编码前创建详细的实现计划

**触发时机：**
- 开始新功能开发
- 进行重大架构变更
- 复杂重构工作
- 需求不明确或模糊时

**工作流程：**

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: 需求重述                                            │
│     - 用清晰术语重新描述需求                                   │
│     - 确认理解是否正确                                         │
├─────────────────────────────────────────────────────────────┤
│  Step 2: 风险识别                                            │
│     - 潜在问题和阻塞点                                         │
│     - 技术债务和依赖风险                                       │
├─────────────────────────────────────────────────────────────┤
│  Step 3: 分阶段计划                                          │
│     - 拆分成具体可执行步骤                                     │
│     - 标注组件间依赖关系                                       │
├─────────────────────────────────────────────────────────────┤
│  Step 4: 复杂度评估                                          │
│     - High/Medium/Low                                        │
│     - 时间估算                                                │
├─────────────────────────────────────────────────────────────┤
│  Step 5: 等待确认                                            │
│     ⚠️ 必须收到用户明确批准才能继续                            │
└─────────────────────────────────────────────────────────────┘
```

**使用示例：**

```
/ecc:plan "添加 OAuth 用户认证"

→ 输出：
  # 实现计划: OAuth 用户认证

  ## 需求重述
  - 支持 Google/GitHub OAuth 登录
  - JWT token 存储在 httpOnly cookie
  - 保护 API 路由

  ## 实现阶段
  Phase 1: 数据库 Schema (用户表、session表)
  Phase 2: OAuth 服务层
  Phase 3: API 路由集成
  Phase 4: 前端登录组件

  ## 风险
  - HIGH: Cookie 跨域问题
  - MEDIUM: Token 过期刷新策略

  ## 复杂度: MEDIUM
  估算: 8-12 小时

  ⚠️ 等待确认: 是否继续？
```

### 2. `search-first` skill - 研究优先工作流

**用途：** 系统化地搜索现有解决方案，避免重复造轮子

**核心理念：** 任何编码前，先搜索是否已有现成方案

**工作流程：**

```
┌─────────────────────────────────────────────┐
│  0. 工具可用性预检                           │
│     检查搜索渠道是否可用                     │
│     诚实报告跳过的渠道                       │
├─────────────────────────────────────────────┤
│  1. 需求分析                                │
│     定义所需功能                             │
│     识别语言/框架约束                        │
├─────────────────────────────────────────────┤
│  2. 并行搜索                                │
│     ┌──────────┐ ┌──────────┐ ┌──────────┐  │
│     │ npm/PyPI │ │ MCP/Skills│ │ GitHub   │  │
│     └──────────┘ └──────────┘ └──────────┘  │
├─────────────────────────────────────────────┤
│  3. 评估候选                                │
│     评分维度：功能、维护、社区、文档、许可    │
├─────────────────────────────────────────────┤
│  4. 决策                                    │
│     ┌─────────┐  ┌──────────┐  ┌─────────┐  │
│     │ 直接采用 │  │ 扩展封装 │  │ 自行开发 │  │
│     └─────────┘  └──────────┘  └─────────┘  │
├─────────────────────────────────────────────┤
│  5. 实施                                    │
│     安装包 / 配置 MCP / 编写最小自定义代码   │
└─────────────────────────────────────────────┘
```

**决策矩阵：**

| 信号 | 行动 |
|------|------|
| 完全匹配、维护良好、MIT/Apache 许可 | **直接采用** — 安装使用 |
| 部分匹配、基础良好 | **扩展封装** — 安装 + 写薄包装层 |
| 多个弱匹配 | **组合使用** — 结合 2-3 个小包 |
| 无合适方案 | **自行开发** — 但基于研究结果 |

**快速模式（内联）：**

```
0. 仓库中已存在？ → rg 搜索相关模块/测试
1. 常见问题？ → 搜索 npm/PyPI
2. 有 MCP 支持？ → 检查 ~/.claude/settings.json
3. 有现成 Skill？ → 检查 ~/.claude/skills/
4. GitHub 有实现？ → 代码搜索维护良好的 OSS
```

**实际案例：**

| 需求 | 搜索 | 发现 | 决策 | 结果 |
|------|------|------|------|------|
| Markdown 死链接检查 | npm "markdown dead link checker" | textlint-rule-no-dead-link (9/10) | 直接采用 | 零自定义代码 |
| HTTP 客户端带重试 | npm "http client retry" | got + retry 插件 | 直接采用 | 生产级库 |
| 配置文件 linter | npm "config linter schema" | ajv-cli (8/10) | 采用+扩展 | 1包+1schema |

**反模式警告：**

| 反模式 | 问题 |
|--------|------|
| 直接写代码 | 不检查是否已有方案 |
| 忽略 MCP | 不检查 MCP server 是否已提供能力 |
| 隐式跳过 | 搜索渠道不可用却不报告 |
| 过度封装 | 包装库太厚，失去原有好处 |
| 依赖膨胀 | 为小功能安装大包 |

### 3. `codebase-onboarding` skill - 项目入职分析

**用途：** 系统化分析陌生代码库，生成入职指南

**触发时机：**
- 第一次打开项目
- 加入新团队/仓库
- 用户问"帮我理解这个代码库"
- 生成 CLAUDE.md 文件

**四阶段工作流：**

#### Phase 1: 侦察

并行收集原始信号：

```
1. 包管理器检测
   → package.json, go.mod, Cargo.toml, pyproject.toml,
     pom.xml, build.gradle, Gemfile, composer.json

2. 框架指纹识别
   → next.config.*, nuxt.config.*, angular.json, vite.config.*,
     django settings, flask app factory, fastapi main

3. 入口点识别
   → main.*, index.*, app.*, server.*, cmd/, src/main/

4. 目录结构快照
   → 前 2 层目录树，忽略 node_modules, vendor, .git

5. 配置和工具检测
   → .eslintrc*, .prettierrc*, tsconfig.json, Makefile,
     Dockerfile, docker-compose*, .github/workflows/

6. 测试结构检测
   → tests/, test/, __tests__/, *_test.go, *.spec.ts
```

#### Phase 2: 架构映射

从侦察数据识别：

- **技术栈**: 语言、框架、数据库、ORM、构建工具、CI/CD
- **架构模式**: 单体/单仓库/微服务/无服务器、前后端分离、API 风格
- **关键目录**: 映射顶层目录到用途
- **数据流**: 追踪请求从入口到响应

#### Phase 3: 约定检测

识别代码库遵循的模式：

- **命名约定**: 文件命名、组件命名、测试文件命名
- **代码模式**: 错误处理、依赖注入、状态管理、异步模式
- **Git 约定**: 分支命名、提交消息风格、PR 工作流

#### Phase 4: 生成入职产物

**产物 1: 入职指南**

```markdown
# 入职指南: [项目名]

## 概述
[2-3 句: 项目做什么，服务谁]

## 技术栈
| 层 | 技术 | 版本 |
|----|----|----|
| 语言 | TypeScript | 5.x |
| 框架 | Next.js | 14.x |

## 关键入口点
- API routes: src/app/api/
- UI pages: src/app/(dashboard)/
- Database: prisma/schema.prisma

## 常见任务
- 运行开发服务器: npm run dev
- 运行测试: npm test
```

**产物 2: CLAUDE.md**

项目特定的 Claude Code 指令文件。

**最佳实践：**

1. 不要读所有文件 — 侦察用 Glob 和 Grep
2. 验证而非猜测 — 代码与配置冲突时信任代码
3. 尊重现有 CLAUDE.md — 增强而非替换
4. 保持简洁 — 2 分钟内可扫描
5. 标记未知 — 无法确定时诚实说明

### 4. `deep-research` skill - 多源深度研究

**用途：** 从多个网络来源生成带引用的研究报告

**触发时机：**
- 用户要求深度研究任何主题
- 竞品分析、技术评估、市场规模
- 公司、投资者、技术的尽职调查
- 用户说"研究"、"深入调查"、"当前状态"

**MCP 要求：**

至少配置一个：
- **firecrawl** — `firecrawl_search`, `firecrawl_scrape`
- **exa** — `web_search_exa`, `web_search_advanced_exa`

**工作流程：**

```
Step 1: 理解目标
  → 澄清 1-2 个问题

Step 2: 规划研究
  → 拆分为 3-5 个研究子问题

Step 3: 多源搜索
  → 对每个子问题搜索
  → 目标 15-30 个独特来源
  → 优先级: 学术/官方/权威新闻 > 博客 > 论坛

Step 4: 深读关键来源
  → 获取 3-5 个关键 URL 完整内容

Step 5: 综合报告
  → 执行摘要
  → 主要主题（带引用）
  → 关键要点
  → 来源列表

Step 6: 交付
  → 短主题: 聊天发布
  → 长报告: 保存文件
```

**质量规则：**

| 规则 | 说明 |
|------|------|
| 每个声明需来源 | 无无来源断言 |
| 交叉验证 | 单一来源时标记为未验证 |
| 时效性 | 优先近 12 个月来源 |
| 承认缺口 | 找不到好信息时说明 |
| 不幻觉 | 不确定时说"数据不足" |

### 5. `documentation-lookup` skill - API 文档查询

**用途：** 通过 Context7 MCP 获取最新库/框架文档

**触发时机：**
- 用户问设置/配置问题
- 请求依赖库的代码
- 需要 API 或参考信息
- 提及特定框架或库

**工作流程：**

```
Step 1: 解析 Library ID
  → resolve-library-id(libraryName, query)
  → 获得 Context7 兼容 ID (如 /vercel/next.js)

Step 2: 选择最佳匹配
  → 名称匹配
  → Benchmark score (文档质量)
  → Source reputation
  → 版本特定 ID

Step 3: 查询文档
  → query-docs(libraryId, query)
  → 限制: 每问题最多 3 次调用

Step 4: 使用文档
  → 回答问题
  → 包含代码示例
  → 引用版本
```

## 综合使用流程

### 典型需求理解流程

```
┌─────────────────────────────────────────────────────────────┐
│ 用户需求: "添加用户认证功能"                                  │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 1: 项目入职 (如果是新项目)                              │
│  → codebase-onboarding skill                                │
│  → 输出: 技术栈、架构模式、关键目录、约定                     │
│                                                              │
│  Step 2: 实现规划                                            │
│  → /ecc:plan "添加用户认证"                                  │
│  → 输出: 需求重述、风险识别、分阶段计划                       │
│  → ⚠️ 等待用户确认                                           │
│                                                              │
│  Step 3: 研究现有方案                                        │
│  → search-first skill                                       │
│  → 搜索: npm/PyPI, MCP servers, Skills, GitHub              │
│  → 输出: 候选评估、决策建议                                   │
│                                                              │
│  Step 4: 查询最新文档                                        │
│  → documentation-lookup skill                               │
│  → 输出: 最新 API 示例代码                                    │
│                                                              │
│  Step 5: 深度研究 (复杂需求)                                  │
│  → deep-research skill                                      │
│  → 输出: 多源综合报告 + 来源引用                              │
│                                                              │
│  最终输出: 完整的认证实现蓝图                                 │
│  - 技术栈选择                                                │
│  - 现有方案评估                                              │
│  - 实现步骤                                                  │
│  - 风险和依赖                                                │
│  - 最新文档参考                                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## Skill 协同关系

| Skill | 与其他 Skill 的协同 |
|-------|---------------------|
| `/ecc:plan` | planner 调用 researcher → 研究结果融入计划 |
| `search-first` | 为 planner/architect 提供可用工具信息 |
| `codebase-onboarding` | 为 planner 提供项目结构背景 |
| `deep-research` | 与 `iterative-retrieval` 组合渐进发现 |
| `documentation-lookup` | 为 search-first 提供最新 API 信息 |

---

## 关键建议总结

| 原则 | 流程 |
|------|------|
| 永远先规划再编码 | `/ecc:plan` → planner → blueprint → 再动手 |
| 永远先研究再编码 | `search-first` → `documentation-lookup` → 再编码 |
| 新项目先入职 | `codebase-onboarding` → 理解技术栈和约定 → 再规划 |
| 复杂需求深度研究 | `deep-research` → 多源综合 → 来源引用 → 再决策 |
| 使用最新文档 | `documentation-lookup` → Context7 → 最新 API |
