技巧和窍门

某些 MCP 是可替代的，可以释放你的上下文窗口

对于 MCP 如版本控制（GitHub）、数据库（Supabase）、部署（Vercel, Railway）等 —— 这些平台中的大多数已经有强大的 CLI，MCP 本质上只是包装它们。MCP 是一个不错的包装，但有代价。

为了让 CLI 功能更像 MCP 而实际上不使用 MCP（以及随之而来的上下文窗口减少），考虑将功能捆绑到技能和命令中。剥离 MCP 暴露的使操作变得简单的工具，并将它们变成命令。

示例：不要始终加载 GitHub MCP，创建一个包装了 gh pr create 及你首选选项的 /gh-pr命令。不要让 Supabase MCP 消耗上下文，创建直接使用 Supabase CLI 的技能。

使用延迟加载，上下文窗口问题基本解决了。但代币使用和成本并没有以同样的方式解决。CLI + 技能方法仍然是一种代币优化方法。

重要内容

上下文和记忆管理

对于跨会话共享记忆，一个总结并检查进度的技能或命令，然后在你的 .claude 文件夹中保存到 .tmp 文件并在会话结束前追加到它是最好的选择。第二天它可以使用那个作为上下文并从你停下的地方继续，为每个会话创建一个新文件，这样你就不会将旧的上下文污染到新工作中。

会话存储示例 -> https://github.com/affaan-m/everything-claude-code/tree/main/examples/sessions

Claude 创建一个总结当前状态的文件。审查它，如有需要请求编辑，然后开始新的对话。只需提供文件路径。这在当你达到上下文限制并需要继续复杂工作时特别有用。这些文件应该包含：

  • 什么方法是有效的（有证据验证）
  • 哪些方法被尝试但没有成功
  • 哪些方法还没有尝试以及剩下什么要做的

策略性清除上下文：

一旦你设置好计划并清除了上下文（Claude Code 现在计划模式中的默认选项），你就可以从计划开始工作。这在你积累了大量不再与执行相关的探索上下文时很有用。对于战略性压缩，禁用自动压缩。在逻辑间隔手动压缩，或创建一个为你执行此操作的技能。

高级：动态系统提示注入

我学到一个模式：不要把所有东西都放在 CLAUDE.md（用户作用域）或 .claude/rules/（项目作用域）中（在每个会话中加载），使用 CLI 标志动态注入上下文。

claude --system-prompt "$(cat memory.md)"

这让你对加载什么的上下文更有针对性。系统提示内容比用户消息有更高权威，用户消息比工具结果有更高权威。

实用设置：

# 日常开发
alias claude-dev='claude --system-prompt "$(cat ~/.claude/contexts/dev.md)"'

# PR 审查模式
alias claude-review='claude --system-prompt "$(cat ~/.claude/contexts/review.md)"'

# 研究/探索模式
alias claude-research='claude --system-prompt "$(cat ~/.claude/contexts/research.md)"'

高级：记忆持久化钩子

大多数人不知道有一些钩子可以帮助记忆：

  • PreCompact 钩子：在上下文压缩发生前，将重要状态保存到文件
  • Stop 钩子（会话结束）：会话结束时，将学习内容持久化到文件
  • SessionStart 钩子：新会话时，自动加载之前的上下文
我已经构建了这些钩子，它们在仓库中：github.com/affaan-m/everything-claude-code/tree/main/hooks/memory-persistence

持续学习 / 记忆

如果你不得不重复一个提示多次，而 Claude 遇到了同样的问题或给了你以前听过的回应 —— 这些模式必须追加到技能中。

问题：浪费的代币、浪费的上下文、浪费的时间。

解决方案：当 Claude Code 发现一些不简单的东西 —— 调试技术、变通方法、一些项目特定的模式 —— 它将那个知识保存为新技能。下次类似问题出现时，技能会自动加载。

我构建了一个执行此操作的持续学习技能：github.com/affaan-m/everything-claude-code/tree/main/skills/continuous-learning

为什么是 Stop 钩子（不是 UserPromptSubmit）：

关键设计决策是使用 Stop 钩子而不是 UserPromptSubmit。UserPromptSubmit 在每条消息上运行 —— 给每个提示增加延迟。Stop 在会话结束时运行一次 —— 轻量级，不会使你的会话变慢。

代币优化

主要策略：子代理架构

优化你使用的工具和子代理架构，委托尽可能便宜但足以完成任务的模型。

模型选择快速参考：

子代理在各种常见任务上的假设设置及选择理由

任务类型 模型 原因
探索/搜索 Haiku 快速、便宜、足以找到文件
简单编辑 Haiku 单文件更改、清晰指令
多文件实现 Sonnet 编码最佳平衡
复杂架构 Opus 需要深度推理
PR 审查 Sonnet 理解上下文，捕捉细微差别
安全分析 Opus 不能遗漏漏洞
写文档 Haiku 结构简单
调试复杂 bug Opus 需要将整个系统记在脑中

默认情况下，对 90% 的编码任务使用 Sonnet。升级到 Opus 当首次尝试失败、任务跨越 5+ 个文件、架构决策、或安全关键代码时。

工具特定优化：

用 mgrep 替换 grep —— 与传统 grep 或 ripgrep 相比平均减少约 50% 的代币：

在我们的 50 任务基准测试中，mgrep + Claude Code 与基于 grep 的工作流相比，使用约 2 倍更少的代币，且质量相当或更好。来源：mgrep by @mixedbread-ai

模块化代码库的好处：

拥有一个更模块化的代码库，主文件在几百行而不是几千行，既有助于代币优化成本，也能在第一次尝试正确完成任务。

验证循环和评估

基准测试工作流：

比较使用技能和不使用技能请求相同内容并检查输出差异：

分叉对话，在其中一个启动一个新的没有技能的工作tree，在结束时拉起 diff，看看记录了什么。

评估模式类型：

  • 基于检查点的评估：设置明确的检查点，根据定义的标准验证，在继续之前修复
  • 持续评估：每 N 分钟或重大更改后运行，完整测试套件 + lint

关键指标：

pass@k: 至少 k 次尝试中有一次成功
        k=1: 70%  k=3: 91%  k=5: 97%

pass^k: 所有 k 次尝试都必须成功
        k=1: 70%  k=3: 34%  k=5: 17%

当只需要它能工作时使用 pass@k。当一致性至关重要时使用 pass^k。

并行化

在多 Claude 终端设置中分叉对话时，确保分叉和原始对话中的操作范围定义良好。在代码更改方面尽量减少重叠。

我的首选模式：

主聊天用于代码更改，分叉用于关于代码库及其当前状态的问题，或外部服务的研究。

关于任意终端数量：

Boris（Anthropic）关于运行多个 Claude 实例

Boris 有并行化的技巧。他建议过在本地运行 5 个 Claude 实例和 5 个上游实例。我建议不要设置任意的终端数量。终端的添加应该是出于真正的必要性。

你的目标应该是：用最少的可行并行化你能完成多少工作。

用于并行实例的 Git Worktrees：

# 为并行工作创建 worktrees
git worktree add ../project-feature-a feature-a
git worktree add ../project-feature-b feature-b
git worktree add ../project-refactor refactor-branch

# 每个 worktree 有自己的 Claude 实例
cd ../project-feature-a && claude

如果你开始扩展你的实例 AND 你有多个 Claude 实例在处理相互重叠的代码，你必须使用 git worktrees 并且对每个都有非常明确的计划。使用 /rename <name here> 来命名你所有的聊天。

初始设置：左终端用于编码，右终端用于问题 - 使用 /rename 和 /fork

级联方法：

当运行多个 Claude Code 实例时，用"级联"模式组织：

  • 在新标签中向右打开新任务
  • 从左到右扫过，从最旧到最新
  • 一次专注于最多 3-4 个任务

代理和子代理的最佳实践

子代理上下文问题：

子代理的存在是为了通过返回摘要而不是转储所有东西来节省上下文。但编排器有子代理缺乏的语义上下文。子代理只知道字面查询，不知道请求背后的目的。

迭代检索模式：

  1. 编排器评估每个子代理返回
  2. 在接受之前问后续问题
  3. 子代理回到源头，获取答案，返回
  4. 循环直到足够（最多 3 个周期）

关键：传递客观上下文，而不仅仅是查询。

带顺序阶段的编排器：

阶段 1: 研究 (使用 Explore 代理) → research-summary.md
阶段 2: 计划 (使用 planner 代理) → plan.md
阶段 3: 实现 (使用 tdd-guide 代理) → 代码更改
阶段 4: 审查 (使用 code-reviewer 代理) → review-comments.md
阶段 5: 验证 (如需要使用 build-error-resolver) → 完成或循环回

关键规则：

  1. 每个代理获得一个明确的输入并产生一个明确的输出
  2. 输出成为下一阶段的输入
  3. 永远不要跳过阶段
  4. 在代理之间使用 /clear
  5. 将中间输出存储在文件中
