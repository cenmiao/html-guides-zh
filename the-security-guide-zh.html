AI代理安全速查指南

开源代理的广泛采用已经到来。提示注入不再是一些愚蠢的模型失败；在代理系统中，它可能变成 shell 执行、秘密暴露、工作流滥用或隐蔽的横向移动。

攻击向量 / 攻击面

攻击向量本质上是任何交互的入口点。你的代理连接的服务越多，你积累的风险就越大。

Simon Willison 的致命三要素框架：私有数据、不可信内容和外部通信。一旦这三者都存在于同一个运行时，提示注入就不再有趣，而开始变成数据泄露。

Claude Code CVE（2026年2月）

CVE-2025-59536。 项目包含的代码可以在信任对话框被接受之前运行。

CVE-2026-21852。 攻击者控制的项目可以覆盖 ANTHROPIC_BASE_URL，重定向 API 流量，在信任确认之前泄露 API 密钥。

沙箱化

Root 访问是危险的。广泛的本地访问是危险的。答案是隔离。

原则很简单：如果代理被入侵，爆炸半径需要很小。

最低标准检查清单

⚠️ 如果你在 2026 年自主运行代理，这是最低标准：

  • ✓ 将代理身份与你的个人账户分离
  • ✓ 使用短期范围凭据
  • ✓ 在容器、devcontainer、VM 或远程沙箱中运行不可信工作
  • ✓ 默认拒绝出站网络
  • ✓ 限制从包含秘密的路径读取
  • ✓ 在有权限的代理看到之前清理文件、HTML、截图和链接内容
  • ✓ 要求批准非沙箱 shell、出口、部署和仓库外写入
  • ✓ 记录工具调用、批准和网络尝试
  • ✓ 实现进程组终止和基于心跳的死人开关
  • ✓ 保持持久记忆狭窄且可丢弃
  • ✓ 像任何其他供应链工件一样扫描技能、钩子、MCP 配置和代理描述符

如果你想要一条规则：永远不要让便利层超过隔离层。
